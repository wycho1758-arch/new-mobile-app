# Plan: user-understandable pod bootstrap reporting

## Scope

Improve `mobile-app-dev-team/09-pod-native-openclaw-skills` so pod agents do not
report raw blocker names such as `git-identity-missing` and
`github-auth-unavailable` without explanation. The skill instructions must guide
agents to explain:

- what the current setup state means;
- which actions the agent already attempted or can complete with local
  CLI/browser/computer-use/MCP tools;
- the smallest non-secret value or human action still needed from the user;
- how the agent will continue after that input exists.

This follow-up will be added to the currently open PR branch
`fix/project-bootstrap-git-github-readiness` / PR #9 after plan approval.

## Problem Statement

The current report and guide are technically correct but can still lead to poor
user-facing output like:

```text
pod role readiness is blocked:
- git-identity-missing
- github-auth-unavailable
```

That is not enough for a user to know what to provide or what the agent can do
next. The required end state is that pod agents can translate blockers into a
small, actionable, secret-safe request, while still doing all agent-owned local
work themselves through CLI/browser/computer-use/MCP tools.

## Planned Changes

1. Add a user-facing result contract to `project-bootstrap/SKILL.md`.
   - Require every blocked final report to include:
     - current state in plain language;
     - agent-owned actions already attempted;
     - remaining minimum user input/action;
     - exact next agent step after input exists;
     - secret-safety warning when auth is involved.
   - Require the agent to say that `/workspace/state/pod-role-bootstrap-report.json`
     is generated by `pod-role-bootstrap`, while `project-bootstrap` reads and
     surfaces it.

2. Make the generated blocker artifact user-understandable in
   `project-bootstrap/scripts/project-bootstrap-preflight.sh`.
   - The generated `/workspace/state/project-bootstrap-blockers.md` must not stop
     at raw blocker names under `Detected Blockers`.
   - Add a synthesized `User-understandable result` section that states:
     - what happened in plain language;
     - which blocker names were detected as supporting status;
     - what the agent already checked or can do with local CLI/browser/computer-use
       and MCP tools;
     - the minimum user request/action;
     - what the agent will do next after the input exists.
   - For `pod-role-bootstrap blocked` with nested `git-identity-missing` and
     `github-auth-unavailable`, generate the specific message that the pod role
     readiness report exists but is still blocked, Git identity needs one approved
     non-secret pair, and GitHub auth needs either approved mounted/managed auth or
     a human-present `gh auth login` / browser/device login flow.
   - Keep raw blocker names only as a secondary technical detail, not as the
     primary user report.

3. Add a reusable human-readable blocker table to
   `project-bootstrap/references/blocker-resolution-guide.md`.
   - For `git-identity-missing`, the user request must be limited to one approved
     non-secret name/email pair or an approved local handoff path. The agent must
     then configure git and rerun bootstrap.
   - For `github-auth-unavailable`, the user request must avoid asking for tokens
     in chat. Acceptable minimum actions are: confirm an approved mounted/managed
     auth source exists, or be present for `gh auth login` / browser/device login.
     The agent may open the browser/tool flow when the human is present and then
     rerun status checks.
   - For nested `pod-role-bootstrap blocked`, explain that the pod report is the
     current readiness result and that `project-bootstrap` is surfacing it, not
     claiming the pod is ready.

4. Add a concise report-template section to
   `project-bootstrap/references/report-template.md`.
   - Document the expected generated blocker Markdown shape, including
     `User-understandable result`, `What the agent already checked`, `Minimum user
     request`, and `Next agent action`.
   - Document an optional `user_summary` shape if future JSON report emission is
     added, but this follow-up must at minimum update the generated Markdown
     artifact.
   - Keep report content secret-safe and status-only.

5. Update `pod-role-bootstrap/SKILL.md` to require human-readable blocker
   translation in its completion/error guidance.
   - It should not ask users to create report files or run status commands.
   - It should tell agents to request only approved Git identity pair or
     human-present GitHub auth action when those exact blockers remain.

6. Update `mobile-app-dev-team/09-pod-native-openclaw-skills/README.md`.
   - Add folder-level guidance that normal pod setup starts from
     `project-bootstrap` and that blocked results must be translated into a
     user-understandable result with minimum request/action.
   - Link or name `project-bootstrap/references/blocker-resolution-guide.md` as
     the place where pod agents find the detailed translation table.

7. Update `mobile-app-dev-team/16-pod-environment-bootstrap.md`.
   - Add an example user-facing minimal request for the common two-blocker case:
     `git-identity-missing` plus `github-auth-unavailable`.
   - Clarify that GitHub connection can be completed by the agent through
     browser/computer-use only when the user is present for credential entry.

8. Add validator/eval coverage before docs/source implementation.
   - Extend `scripts/validate-team-doc.mjs` assertions for the new required terms:
     `user-understandable result`, `minimum user request`, `agent can continue`,
     `browser/device login with human present`, and the pod-role report ownership
     distinction.
   - Extend or add smoke eval assertions that the generated
     `/workspace/state/project-bootstrap-blockers.md` for the nested
     `pod-role-bootstrap blocked` case contains:
     - `User-understandable result`;
     - plain-language pod readiness blocked text;
     - `Minimum user request`;
     - approved non-secret Git identity pair wording;
     - human-present `gh auth login` or browser/device login wording;
     - next agent action wording.
