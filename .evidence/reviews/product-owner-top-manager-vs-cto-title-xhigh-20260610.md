**Recommendation**

Yes, it is acceptable to make Product/Planning the top management/coordinator for intake, scope, readiness, work routing, evidence expectations, and human-gate management. The clean title is **Product Manager / Product Owner**. If the user wants a stronger “top of work pyramid” label, use **Product Delivery Lead** or **Product Owner / Delivery Lead**, while keeping the operating role exactly `Product/Planning`.

Do not name Product/Planning `CTO`.

For Mobile Architect, prefer **Mobile Technical Lead** or **Mobile Architect / Technical Lead**. Avoid `CTO` entirely in runtime-facing files unless a future SoT explicitly defines it as a human-only display alias and validators are updated to distinguish display title from operating role. Current SoT says `CTO` is not a runtime role.

Clean split:

| Position title | Operating Role | Authority |
| --- | --- | --- |
| Product Manager / Product Owner, or Product Delivery Lead | Product/Planning | Scope, readiness, acceptance criteria, evidence expectations, human-gate routing |
| Mobile Technical Lead | Mobile Architect | Architecture, route/state, runtime policy, API contract co-review, releaseability risk |
| Product Designer | Design | UX quality and Stitch handoff |
| Mobile App Developer | Mobile App Dev | Expo RN implementation and tests |
| Backend/API Engineer | Backend/API Integrator | Contracts, schemas, backend/API integration |
| QA/Release Engineer | QA/Release | Evidence, E2E, release readiness reporting |
| Release Gatekeeper | Gatekeeper | Deterministic pass/fail only, no SOUL |
