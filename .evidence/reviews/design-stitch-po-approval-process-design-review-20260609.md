**Findings**

1. **High: P0/P1 gates are not yet Product/Planning-owned states.**  
   The proposed return tokens `APPROVED_FOR_STITCH_GENERATION` and `APPROVED_FOR_HTML_EXTRACTION` do not map to current `po-*` outputs. Existing Product/Planning readiness states are things like `READY_FOR_EXECUTION`, `NEEDS_REWORK`, `HUMAN_DECISION_REQUIRED`, and `BLOCKED_BY_RUNTIME_CAPABILITY`, not Stitch-specific approvals. Without a PO-side contract or accepted SoT record, Design would be asking for approvals no Product/Planning skill is currently defined to issue. Sources: [po-planning-completeness-review](</Users/tw.kim/Documents/AGA/test/new-mobile-app/.agents/skills/po-planning-completeness-review/SKILL.md:36>), [po-prd-to-execution](</Users/tw.kim/Documents/AGA/test/new-mobile-app/.agents/skills/po-prd-to-execution/SKILL.md:21>), [PROJECT_ENVIRONMENT.md](</Users/tw.kim/Documents/AGA/test/new-mobile-app/PROJECT_ENVIRONMENT.md:186>).

2. **High: The HTML extraction boundary needs stricter tool-level wording.**  
   The proposed “images only first; no HTML extraction yet” is directionally correct, but current Design/Stitch skills fetch HTML immediately after generating options. Google’s `generate-design` skill also downloads HTML and screenshot assets after generation, and the SDK exposes `getHtml()` and `getImage()` as adjacent screen methods. The extension should explicitly forbid `fetch_screen_code`, ZIP `code.html`, SDK `getHtml()`, and any `get_screen`/metadata path that persists `htmlCode.downloadUrl` before P1 approval. Sources: [design-stitch-mcp-operating-rules](</Users/tw.kim/Documents/AGA/test/new-mobile-app/.agents/skills/design-stitch-mcp-operating-rules/SKILL.md:27>), Google Stitch `generate-design` skill ([raw.githubusercontent.com](https://raw.githubusercontent.com/google-labs-code/stitch-skills/main/plugins/stitch-design/skills/generate-design/SKILL.md)), Stitch SDK screen API ([github.com](https://github.com/google-labs-code/stitch-sdk)).

3. **High: DESIGN.md decision is too late if first recorded at P1.**  
   Current repo Design flow requires the `DESIGN.md` decision before Stitch work, and Stitch operating rules apply `DESIGN.md` context before generation. The proposed P1 packet includes the DESIGN.md decision after visuals are generated, which risks generating from an unapproved or stale design-system SoT. P0 must include `KEEP_EXISTING_DESIGN_MD`, `UPDATE_DESIGN_MD_REQUIRED`, or `BLOCKED_BY_DESIGN_SYSTEM_DECISION`; if update/upload is needed, generation should block until confirmed. Google’s manage-design-system skill also requires explicit confirmation before uploading `DESIGN.md`. Sources: [design-mobile-design-handoff](</Users/tw.kim/Documents/AGA/test/new-mobile-app/.agents/skills/design-mobile-design-handoff/SKILL.md:31>), [design-stitch-mcp-operating-rules](</Users/tw.kim/Documents/AGA/test/new-mobile-app/.agents/skills/design-stitch-mcp-operating-rules/SKILL.md:21>), Google manage-design-system skill ([raw.githubusercontent.com](https://raw.githubusercontent.com/google-labs-code/stitch-skills/main/plugins/stitch-design/skills/manage-design-system/SKILL.md)).

4. **Medium: Gemini 3.1 Pro preference should be recorded as requested capability, not assumed capability.**  
   The official basis supports Gemini 3.1 Pro for complex tasks, but the Stitch SDK source surfaced here lists `modelId` values as `GEMINI_3_PRO` and `GEMINI_3_FLASH`, not `3.1`, and the repo MCP contract only pins `stitch-mcp@1.3.2`. The manifest should record requested model/mode, exposed tool capability, actual selected model/mode when returned, and a limitation when the MCP surface does not expose selection. Sources: [PROJECT_ENVIRONMENT.md](</Users/tw.kim/Documents/AGA/test/new-mobile-app/PROJECT_ENVIRONMENT.md:226>), Stitch SDK model IDs ([github.com](https://github.com/google-labs-code/stitch-sdk)), Gemini 3.1 Pro blog ([blog.google](https://blog.google/innovation-and-ai/models-and-research/gemini-models/gemini-3-1-pro/?cmid=c9b01550-0b3f-4378-ac24-2de1381b4590)).

5. **Medium: “Exactly two options” needs multi-screen/state precision.**  
   Current skills require exactly two options and every screen to include default, loading, empty, error, and permission-denied states. The extension should define Option A and Option B as two design directions, each covering all approved screens and five states, rather than two total screens or uncontrolled variants per screen. Sources: [design-mobile-design-handoff](</Users/tw.kim/Documents/AGA/test/new-mobile-app/.agents/skills/design-mobile-design-handoff/SKILL.md:35>), [design-stitch-mcp-operating-rules](</Users/tw.kim/Documents/AGA/test/new-mobile-app/.agents/skills/design-stitch-mcp-operating-rules/SKILL.md:51>).

6. **Medium: Eval/validator work is required, not optional.**  
   Runtime changes under `.agents/`, `evals/`, or scripts must pass runtime/local harness gates, and TDD is required. Current validator assertions check Stitch, DESIGN.md, HTML extraction, exactly two options, images, and publication artifacts, but not P0/P1 approvals or the “no HTML before P1” boundary. Add positive and negative evals for P0 required, P1 required, rejected/NEEDS_REWORK handling, missing approval artifacts, and premature HTML extraction. Sources: [AGENTS.md](</Users/tw.kim/Documents/AGA/test/new-mobile-app/AGENTS.md:7>), [AGENTS.md](</Users/tw.kim/Documents/AGA/test/new-mobile-app/AGENTS.md:84>), [validate-runtime-artifacts.mjs](</Users/tw.kim/Documents/AGA/test/new-mobile-app/scripts/validate-runtime-artifacts.mjs:349>).

**Review Result**

Needs rework before implementation. The direction is sound, especially splitting visual generation from HTML extraction, but the approval states, DESIGN.md timing, and enforcement tests must be made explicit before changing the Design/Stitch skills. No files were modified.
