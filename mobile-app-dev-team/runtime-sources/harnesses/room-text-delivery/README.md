# Room Text Delivery Harness

Room Text Delivery Harness is a small, repo-owned validator for visible Room text
message delivery proof. It exists to prove that a report was visibly delivered to
the intended Room, not to decide whether any broader task, Workboard card, PR, or
workflow is complete.

## Scope

The harness validates a normalized Room text delivery result. It intentionally
stays transport-neutral:

- raw `curl` sends are allowed;
- `bin/send-room-text.mjs` is allowed as a convenience wrapper;
- MCP/tool sends are allowed when they can produce the normalized result shape.

Acceptance is the normalized delivery result plus validator PASS. `message_id`
and `room_id` are delivery proof only and must not be treated as work completion
proof.

## Validate fixtures

```bash
node mobile-app-dev-team/runtime-sources/harnesses/room-text-delivery/validators/validate-room-text-result.mjs --self-test
```

Validate one result:

```bash
node mobile-app-dev-team/runtime-sources/harnesses/room-text-delivery/validators/validate-room-text-result.mjs result.json
```

Expected exit codes:

- `0`: valid Room text delivery proof.
- `1`: invalid, blocked, or non-Room substitute proof.

## Live smoke boundary

Do not perform a live Room smoke unless Product/Planning explicitly approves the
smoke room. Prefer a dedicated test Room. Do not use any Kim/Spring reporting Room unless Kim/Spring
explicitly approves. Live smoke proof must be reported as delivery proof only.

## Acceptance note: plain text alone

Visible Room text without a normalized delivery result and validator PASS is not harness proof. Raw curl, `send-room-text.mjs`, MCP/tool sends, and other transports remain allowed only when their response is normalized and passes the validator.

## Access-boundary note

A 403 from an unauthorized Room send is valid route-boundary evidence and should FAIL delivery proof. It is not by itself an implementation or validator bug.

## Expected destination rule

For report-delivery proof, `send-room-text.mjs` requires an explicit expected destination via `--expected-room-id` or `--visible-report-destination` unless the run is `--dry-run` or explicitly `--result-kind transport-smoke`. This prevents request Room ids from being silently reused as proof destinations.
