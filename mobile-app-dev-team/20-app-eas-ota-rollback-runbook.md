# App / EAS / OTA Rollback Runbook

This is managed-doc guidance (SoT priority 5) and does not supersede higher-priority SoT. Runtime enforcement is largely a deferred follow-up, with one deterministic slice already enforced: a `production-submit` `human-gate/v1` decision must carry `rollback_owner` and `rollback_plan` (see §Gates and Evidence). The remaining rollback governance stays managed-doc/process-owned.

This document is a governance, ownership, decision, gate, and evidence runbook for app code, EAS Update (OTA), and store rollback. Every claim below is grounded in an actual line of a higher-priority SoT file and cites it as `<path>` §<section>. Where the SoT does not define a state, the gap is named explicitly rather than filled. This document does not supersede or claim to be already enforced by `.agents/skills/*` or repo validators.

## Purpose

The repository already defines rollback only as scattered fragments: an approval-envelope `rollback_owner` / `rollback_plan` requirement, an external-rollback ownership rule, a backend rollback-note artifact, and a NativeWind v5 rollback criterion. This runbook completes those fragments into a single ownership / decision / gate / evidence runbook for app, EAS Update (OTA), and store rollback.

This document does not authorize, prove, or imply that live EAS, EAS Update (OTA) channel rollback, store rollback, or production execution may proceed from repo-local evidence. The annex states that it "does not authorize live EAS, Stitch, Google Cloud, mobile-mcp/device, Railway, Confluence, GitHub settings, pod rollout, image build/push, Secret/token provisioning, release readiness, or store submission" (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Purpose). This runbook stays inside that boundary: repo agents document the rollback plan and evidence and classify failures; they do not own or execute external rollback.

## Existing Partial Definition

Four existing fragments define rollback today. This runbook builds only on these and does not extend them beyond what they say.

- Approval-envelope rollback fields. Every live mutation or live-readiness claim must record an approval before execution, and that approval must include `rollback_owner` ("Human or ops owner responsible for undoing the mutation") and `rollback_plan` ("Concrete rollback path or reason rollback is not available") (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Approval Envelope).
- External rollback ownership. "Rollback must be owned by the approving human/ops owner. Repo agents can document rollback evidence and classify failures, but they do not own external rollback." (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Stop And Rollback Rules).
- Backend rollback-note artifact. Backend/API Integrator owns a managed `03-contract-api/rollback-note.md` output (`mobile-app-dev-team/10-github-artifact-workflow.md` §Backend/API Integrator), produced when approved scope includes backend service delivery "with DB schema/migration note, deployment config note, runtime smoke result, rollback note, and service evidence" (`mobile-app-dev-team/05-work-processes.md` §3 API Readiness, step 4).
- NativeWind v5 rollback criteria. The styling layer pins a pre-release `nativewind@5.0.0-preview.4`, with explicit rollback criteria: "Roll back to the last passing NativeWind v4 / Tailwind CSS v3 baseline if SDK 56 compatibility checks, Metro bundling, Jest, native run smoke, or Maestro/mobile visual QA fail because of NativeWind v5 and cannot be fixed in the same PR." and "Do not promote a production release with unresolved NativeWind v5 rendering, bundling, or native runtime defects." (`PROJECT_ENVIRONMENT.md` §Mobile Styling).

## Role Ownership

Rollback ownership follows the existing role capability split. No role absorbs another role's ownership.

- Mobile Architect / Technical Lead owns releaseability, EAS strategy, rollback architecture, and route/state decisions. The capability matrix records that this role can "Decide architecture, route/state impact, EAS strategy" and produces "ADR, risk note, route/state review, contract co-sign", while it must not "Absorb implementation/backend ownership" (`mobile-app-dev-team/03-role-capability-matrix.md` §Role Capability Matrix, Mobile Architect row). For releaseability work, the Architect workflow records "EAS and QA/Release evidence implications without approving release readiness" (`.agents/skills/mobile-architect-workflow/SKILL.md` §Workflow, step 7). The Architect decides the rollback architecture and route/state impact; it does not approve release readiness.
- QA/Release Engineer owns rollback evidence for the EAS Update channel, store, and native surfaces, the release-risk summary, and failure classification. The capability matrix records that QA/Release can "Plan/run evidence and release checks" and produces "E2E evidence, gate classification, release risk summary", and must not "Fix implementation or approve failed gates" (`mobile-app-dev-team/03-role-capability-matrix.md` §Role Capability Matrix, QA/Release row). In the QA and release process, QA/Release records evidence, then "Classify failures and route to owner" (`mobile-app-dev-team/05-work-processes.md` §5 QA And Release Evidence, steps 4-5). QA/Release documents and classifies rollback evidence; it does not execute external rollback or approve a failed gate.
- Human / ops owner owns external rollback EXECUTION and the approval envelope. "Rollback must be owned by the approving human/ops owner." (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Stop And Rollback Rules), and every live mutation requires a recorded approval envelope with `rollback_owner` and `rollback_plan` before execution (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Approval Envelope).
- Product/Planning (Chief Product Officer (CPO) / Product Delivery Lead) owns routing and human gates. This role can "route gates, coordinate role owners" (`mobile-app-dev-team/03-role-capability-matrix.md` §Role Capability Matrix, Product/Planning row), and the process requires routing technical decisions to Mobile Architect when "releaseability risk exists" and routing "human gates before any execution work" (`mobile-app-dev-team/05-work-processes.md` §1 Intake And Planning, steps 6-7).

## Rollback Scenarios

The following are governance flows, not fabricated live procedures. Each names the deciding role, the evidence to re-run or document, and where external execution stays human/ops-owned.

- (a) App code/build rollback before release. Revert to the last passing build or commit. Mobile Architect decides the architecture and route/state impact of the revert (`mobile-app-dev-team/03-role-capability-matrix.md` §Role Capability Matrix, Mobile Architect row). QA/Release re-runs the relevant evidence-ladder level and records the result (`mobile-app-dev-team/06-gates-and-evidence.md` §Mobile Evidence Ladder; `mobile-app-dev-team/05-work-processes.md` §6 Failure Loop, step 4). This is a repo-local decision; it does not by itself prove release readiness.
- (b) NativeWind v5 rollback. Apply the existing PROJECT_ENVIRONMENT.md criteria: roll back to the last passing NativeWind v4 / Tailwind CSS v3 baseline when the listed checks fail because of NativeWind v5 and cannot be fixed in the same PR, and do not promote a production release with unresolved NativeWind v5 defects (`PROJECT_ENVIRONMENT.md` §Mobile Styling).
- (c) EAS Update (OTA) channel rollback. This is human/ops-owned through the approval envelope, using the `rollback_owner` and `rollback_plan` fields (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Approval Envelope). Running `eas` or claiming native readiness is forbidden without approval, and an `EXPO_TOKEN`-based or live EAS claim is not provable from repo-local evidence (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Live Categories And Required Evidence, Live EAS/Maestro row; §Forbidden Claims). Repo agents document the rollback plan and classify failures only (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Stop And Rollback Rules).
- (d) Store rollback / mitigation. This is owned by the human / ops owner together with the store owner. It is `production-submit`-class work requiring a human approval record plus release evidence (`mobile-app-dev-team/06-gates-and-evidence.md` §Required Gates, Production submit row). The repository cannot prove store submission or store rollback; "store-submission safety" is a forbidden repo-local claim (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Forbidden Claims).
- (e) Backend service rollback. Use the existing `03-contract-api/rollback-note.md` artifact owned by Backend/API Integrator (`mobile-app-dev-team/10-github-artifact-workflow.md` §Backend/API Integrator), prepared when approved scope includes backend service delivery (`mobile-app-dev-team/05-work-processes.md` §3 API Readiness, step 4).

## Gates and Evidence

- Record `rollback_owner` and `rollback_plan` in the approval envelope before any live mutation or live-readiness rollback claim (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Approval Envelope).
- **Runtime enforcement (partial)**: for a `human-gate/v1` decision with `category` = `production-submit` (the live production mutation), `rollback_owner` and `rollback_plan` are required non-empty fields, deterministically validated in `scripts/lib/work-unit-machine.mjs` (`validateHumanGateDecision`) and covered by `scripts/validate-work-units.mjs --self-test` fixtures. This binding was authorized by the human owner and scoped by a `po-scope-gate-reviewer` (xhigh) decision at `mobile-app-dev-team/decision-records/20260614-followup1-precise-rule-decision.md`; other human-gate categories keep their existing requirements. This is managed-doc guidance (SoT priority 5) pointing at the validator that enforces it; it does not supersede higher-priority SoT.
- Use the `failed-gate-risk` human gate when accepting residual risk after a failed gate. Stop for a recorded human decision when "Accepting risk after a failed gate" applies, and `failed-gate-risk` is a deterministic human-gate category that additionally requires `failed_check_reference` (`mobile-app-dev-team/06-gates-and-evidence.md` §Human Gates).
- Never bypass `production-submit`. Even expedited emergency hotfixes keep the `production-submit` and `failed-gate-risk` human gates: they "are never bypassed" (`mobile-app-dev-team/06-gates-and-evidence.md` §Human Gates).
- QA/Release records rollback evidence under `.evidence/` or the durable work-unit handoff root. Evidence "should live under `.evidence/`" when persisted proof is required (`mobile-app-dev-team/06-gates-and-evidence.md` §Evidence Rules), and pod-isolated handoff uses `docs/plans/work-units/<work-unit-id>/` as the committed durable GitHub handoff root (`mobile-app-dev-team/06-gates-and-evidence.md` §Durable GitHub Handoff). Work-unit QA files summarize and link canonical evidence; they do not replace it (`mobile-app-dev-team/05-work-processes.md` §5 QA And Release Evidence, step 7).

## Boundary

This runbook documents the ownership, decision, gate, and evidence flow for app, EAS Update (OTA), and store rollback only. It does NOT authorize or prove live EAS Update channel rollback, store rollback, or production execution from repo-local evidence. Live EAS/Maestro, native behavior, release readiness, "production submit approval", and "store-submission safety" are explicitly forbidden as repo-local claims (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Forbidden Claims). External rollback execution remains owned by the approving human/ops owner (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Stop And Rollback Rules). The correct phrasing for any rollback claim is the annex phrasing: "repo-local/offline proof is complete for the named slice; live readiness remains pending human/ops approval and evidence." (`mobile-app-dev-team/15-human-ops-live-readiness-annex.md` §Forbidden Claims).
