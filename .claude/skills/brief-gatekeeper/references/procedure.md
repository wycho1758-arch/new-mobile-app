# Brief Gatekeeper — 절차 상세 (v0.13)

SKILL.md가 개요라면, 이 문서는 각 단계의 *세부 실행 규칙*을 다룬다.

## Contents
- Step 1: 문서·요청 이해
- Step 2: 명확성 체크리스트 판정 (가시 출력 필수)
- Step 3: 질문 규칙 (환경 적응)
- Step 4: 이해 확인 (전 분기 필수)
- Step 5: 6기준 간이 판정 (조건부 필수)
- Step 6: 실행 또는 에스컬레이션 핸드오프
- 멈추는 조건

## Step 1: 문서·요청 이해

**목표:** 문서가 무엇이고 사용자가 무엇을 원하는지 기본 파악.

**출력해야 할 최소 정보:**
- 문서 종류 (데이터/보고서/계약서/대시보드/코드/텍스트 등)
- 도메인 (곡물 트레이딩·해운·재무·HR 등)
- 한 줄 요약

**문서 이해 불가 시:**
1. "이 파일이 어떤 것인지 간단히 알려주시겠어요?" 수준으로 질문.
3. 이 확인은 **질문 라운드 카운트에서 제외**한다.
4. 이해가 완료된 후에만 Step 2로 진행한다.

## Step 2: 명확성 체크리스트 판정 (가시 출력 필수)

**반드시 판정표를 응답에 포함시킨다.** 내부 판정만 하고 숨기면 v0.13 기준 Fail.

### 필수 출력 형식

```
## 명확성 체크리스트 판정
| # | 항목 | 판정 | 사유 |
|---|------|:----:|------|
| 1 | 요청 동사 | ✅ YES / ❌ NO | ... |
| 2 | 집중 범위 | ✅ YES / ❌ NO | ... |
| 3 | 결과물 형태 | ✅ YES / ❌ NO | ... |
| 4 | 배경 정보 | ✅ YES / ❌ NO / ➖ N/A | ... |
```

### #1 요청 동사 — 구체·모호 구분

**구체 동작 (YES):** 분석, 요약, 검토, 작성, 추출, 비교, 계산, 번역, 변환, 교정, 생성, 매핑, 정규화, 분류

**모호 동작 (NO):** 정리, 처리, 봐줘, 손봐줘, 확인해줘, 체크해봐, 이거 해줘, 어떻게 생각해, 조치해줘, 다뤄줘

→ 모호 동작이 오면 **재질문 필수** (Step 3에서 "어떤 작업인지" 묻는다).

### #2 집중 범위 — 판정 기준

- 전체 / 특정 섹션 / 특정 관점 중 하나가 **명시되어야 YES**.
- "이거", "이 문서"만 있고 추가 단서가 없으면 → **NO**. 단, 문서가 단일 주제이고 사용자가 "전체"를 암묵적으로 의미한다고 볼 합리적 근거가 있으면 YES 처리 가능 (사유에 명시).

### #3 결과물 형태 — 판정 기준

요약텍스트 / 표 / 엑셀 / CSV / 보고서(MD or PDF) / 슬라이드 / 차트 / 코드 / 텍스트 답변 — 중 하나가 명시되어야 YES.
"보기 편하게", "깔끔하게" 같은 추상적 표현은 **NO**.

### #4 배경 정보 — 조건부 규칙

**요청 유형이 아래 중 하나면 필수 (YES/NO 판정):**
- 분석 (경쟁사 분석, 추세 분석, 원인 분석 등)
- 전략 제안 (액션 플랜, 로드맵 제안)
- 보고서 작성 (임원용, 대외용 등)
- 평가·검토·리뷰

**그 외 (단순 추출·변환·계산)** → N/A

대상·범위기준·시점·작업배경 중 **2개 이상이 식별되면 YES**, 1개 이하면 NO.

### 진행 조건

```
if 모든 필수(#1~#3) = YES and #4 in (YES, N/A):
    → Step 4 (이해 확인)
else:
    → Step 3 (질문)
```

## Step 3: 질문 규칙

### 질문 포맷

**구조화된 질문 포맷**을 사용한다: 번호 매긴 선택지 + ★ 권장 옵션 표기.

**선택지 4~5개 + 권장 옵션 ★는 필수.**

### 라운드 규칙 (최대 2라운드)

- **라운드 1**: 부족한 *모든* 필수 항목을 한 번에 묶어 질문.
- **라운드 2**: 라운드 1 답변이 불충분하면 부족 항목만 1회 더 질문.
- **2라운드 후에도 불명확 → 멈춤** (분기 5).

### 재질문 금지 규칙 (오용 방지)

- 이미 YES로 판정된 항목은 *다시 묻지 않는다*.
- 그러나 *다른 NO 항목은 반드시 묻는다.* "YES가 하나라도 있으니 전체 스킵"은 v0.13에서 금지.

### 포맷 예시

```
진행 전에 확인할 부분이 있어요.

1. 이 데이터로 어떤 작업을 해드릴까요?
   - (a) 요약
   - (b) 분석 ★
   - (c) 비교
   - (d) 보고서 작성
   - (e) 기타 (직접 입력)

2. 어디에 집중하면 될까요?
   ...
```

## Step 4: 이해 확인 (전 분기 필수)

**예외 없음. 분기 1(바로 진행)·3(역할 위임)·4(에스컬레이션 핸드오프) 어느 경우에도 실행.**

### 분기 1 전용 가드

분기 1(전항목 YES)에서도 Step 4는 반드시 **사용자 응답을 받는 단계**로 실행한다.

**금지 항목:**
- 체크박스만 체크하고 넘어가기 → Fail
- "자명", "명확함", "추가 확인 불필요" 등으로 Step 4 스킵 정당화 → Fail
- Step 4를 완료로 표기(`[x] Step 4`)하려면 사용자 응답(a/b/c)이 선행되어야 함. 응답 없이 표기 → 체크리스트 위조(Fail)
- 동일 턴에 결과물 파일 업로드 → Fail. **Step 4 턴과 실행 턴은 반드시 분리**

### 필수 산출물 3개

Step 4 응답에는 아래 3개가 반드시 포함되어야 한다. 하나라도 누락되면 Fail.

**① 요약 박스:**
```
- 요청: {동작 + 대상}
- 집중 범위: {범위}
- 결과물: {형식 + 분량}
- 배경: {대상·시점·작업배경} (해당 시)
```

**② 진행 방식 리스트:**
```
- 진행 방식: 1) ... 2) ... 3) ...
- 예상 소요: 약 {n분|n시간}
```

**③ 선택지 블록 (필수 — 없으면 자동 Fail):**

선택지는 최소 2개(진행/수정), 권장 옵션(★) 1개 포함.

```
- (a) 맞습니다, 이대로 진행해주세요 ★
- (b) 수정이 필요합니다 (어떤 부분인지 알려주세요)
```

수정 요청 시 해당 항목만 재질문. 이 수정 질문은 **라운드 2로 카운트**한다.

## Step 5: 6기준 간이 판정 (조건부 필수)

### 수행 조건

**다음 중 하나에 해당하면 필수 수행:**
- 분석 요청
- 전략 제안 요청
- 보고서 작성 요청
- 평가·검토·리뷰 요청

**스킵 조건:** 단순 추출·변환·계산·번역·형식 변환.

### 판정 방법

각 기준을 `low(1) / mid(2) / high(3)`로 평가. 응답에 표로 노출.

| 기준 | 판정 | 점수 | 근거 |
|------|:---:|:---:|------|
| 업무 유형 복합도 | low/mid/high | 1~3 | ... |
| 공식 vs 판단 | low/mid/high | 1~3 | ... |
| 외부 커뮤니케이션 | low/mid/high | 1~3 | ... |
| 표준 절차 | low/mid/high | 1~3 | ... |
| 도메인 전문성 | low/mid/high | 1~3 | ... |
| 실수 영향도 | low/mid/high | 1~3 | ... |

상세 기준: [escalation-criteria.md](escalation-criteria.md).

### 호출 조건

둘 중 하나라도 해당 시 Product/Planning 핸드오프:
- `high ≥ 3`
- `총점 ≥ 14` (만점 18)

### insufficiency_reason 생성

high 또는 mid로 판정된 기준의 사유를 한 줄씩 취합하여 `insufficiency_reason` 문자열을 생성한다.
이 문자열은 handoff-schema의 필수 필드로 전달된다.

---

## Step 6: 실행 또는 에스컬레이션 핸드오프

- **기존 역할로 충분** → 그대로 실행 (Step 4 확인이 "맞습니다"인 경우에만)
- **기존 구조로 부족** → Product/Planning 핸드오프 제시 + [handoff-schema.md](handoff-schema.md)에 따라 인자 전달

### 핸드오프 보충 요청 대응

1. 반려 응답에서 부족 필드명 + 사유 수신
2. 해당 필드만 보충하여 **1회 재전달**
3. **2회 반려 시 멈춤** (사용자에게 상황 안내)

---

## 멈추는 조건

다음 상황에서는 진행하지 않는다:
- Step 1에서 문서 이해 불가 + 추가 확인 후에도 불가
- Step 3 2라운드 후에도 핵심 불명확
- Step 4 확인에서 수정 후에도 합의 불가
- 핸드오프 보충 요청 2회

**애매한 상태를 덮고 넘어가지 않는다.**
